## 项目概述
- **名称**: 论坛体小说生成工作流
- **功能**: 根据用户输入的主题自动生成完整的论坛体小说，支持开放多元的结局类型和吸睛标题

### 节点清单
| 节点名 | 文件位置 | 类型 | 功能描述 | 分支逻辑 | 配置文件 |
|-------|---------|------|---------|---------|---------|
| outline_node | `nodes/outline_node.py` | agent | 生成故事大纲 | - | `config/outline_llm_cfg.json` |
| opening_node | `nodes/opening_node.py` | agent | 生成吸睛标题和开篇帖子（1楼） | - | `config/opening_llm_cfg.json` |
| replies_node | `nodes/replies_node.py` | agent | 生成网友回复（2-40楼，跳过11楼、21楼、31楼、41楼） | - | `config/replies_llm_cfg.json` |
| update_node | `nodes/update_node.py` | agent | 生成楼主回复（11楼、21楼、31楼、41楼） | - | `config/update_llm_cfg.json` |
| merge_node | `nodes/merge_node.py` | task | 整合所有楼层生成完整故事 | - | - |

**类型说明**: task(task节点) / agent(大模型) / condition(条件分支) / looparray(列表循环) / loopcond(条件循环)

## 工作流结构
- **主图**: 线性流程
  1. outline_node（生成故事大纲）
  2. opening_node（生成吸睛标题和开篇帖子，1楼）
  3. replies_node（生成网友回复，2-40楼，跳过11楼、21楼、31楼、41楼）
  4. update_node（生成楼主回复，11楼、21楼、31楼、41楼）
  5. merge_node（整合所有楼层）

## 核心特性
- **吸睛标题**：生成5-20字的吸引眼球的标题，有悬念感，网感十足，切中痛点
- **楼层号连续**: 生成2-41楼的完整楼层序列，不跳层
- **楼主回复分散**: 11楼、21楼、31楼、41楼生成楼主回复，结局内容分散到多楼，更有层次感
- **结局类型开放多元**: 不局限于固定模式，支持甜宠HE、悲剧BE、开放式、悬疑反转、现实向、成长向、遗憾美、暧昧向、惊悚向、黑暗向、温情向、苦尽甘来、得不偿失、因果报应、机缘巧合、自我救赎、意料之外情理之中等任何符合故事逻辑的结局类型
- **结局有深度**: 结局符合人物性格和现实逻辑，有思考价值，能够引发读者共鸣或反思
- **结局内容分散**: 把结局内容分散到21楼、31楼、41楼，每楼50-100字，简洁有力，不要都挤在最后一楼
- **开篇格式灵活**: 支持楼主发帖、直接分享等多种格式
- **内容质量高**: 每条回复都有独特价值，不能水贴
- **语气真实**: 使用网络流行语，像真人发的

## 楼层布局
- **1楼**: 开篇帖子
- **2-10楼**: 建立氛围，制造槽点，埋下伏笔
- **11楼**: 楼主回复（回应网友，埋下伏笔）
- **12-20楼**: 推进剧情，多角度分析
- **21楼**: 楼主回复（表达内心状态，开始铺垫结局）
- **22-30楼**: 高潮迭起，为结局做铺垫
- **31楼**: 楼主回复（推进到结局阶段，揭示关键信息或转折）
- **32-40楼**: 推动故事走向结局，给楼主发言空间
- **41楼**: 楼主回复（最终的结局或悬念）

## 标题生成规则
- **吸睛原则**：标题必须吸引眼球，让人有点击欲望
- **悬念感**：用疑问句、感叹句、数字等制造悬念
- **简洁有力**：5-20字，不要太长
- **网感十足**：使用网络流行语、反差感、夸张手法
- **切中痛点**：抓住读者的好奇心和八卦心理
- **标题类型参考**：
  - 疑问式：「救命，我做错了吗？」
  - 数字式：「昨天做了3件事，现在后悔了」
  - 对比式：「他对我那么好，我却拉黑了他」
  - 悬念式：「发现室友的秘密，我该怎么办」
  - 夸张式：「社死现场，我还能抢救吗」
  - 反转式：「本来想骂他，结果...」
  - 情感式：「心凉了，他居然这样做」
  - 呼救式：「家人们谁懂啊，救救我」

## 结局类型说明
本工作流不局限任何结局类型，而是根据故事主题、人物性格、现实逻辑，自然发展出符合故事的结局。可能的结局类型包括但不限于：
- 甜宠HE（大团圆）
- 悲剧BE（遗憾、失败）
- 开放式（留白、想象空间）
- 悬疑反转（意想不到）
- 现实向（平凡、真实）
- 成长向（自我认知、成长）
- 遗憾美（错过、释怀）
- 暧昧向（不明说、留悬念）
- 惊悚向（细思极恐）
- 黑暗向（人性复杂）
- 温情向（治愈、温暖）
- 苦尽甘来（先苦后甜）
- 得不偿失（得到但失去）
- 因果报应（恶有恶报）
- 机缘巧合（命运安排）
- 自我救赎（弥补过错）
- 意料之外情理之中
- 或任何其他符合逻辑的结局类型

## 技能使用
- 所有节点均使用大语言模型技能（coze-coding-dev-sdk）

## 配置文件说明
- **outline_llm_cfg.json**: 故事大纲生成配置（结局方向开放多元）
- **opening_llm_cfg.json**: 开篇帖子生成配置（吸睛标题、灵活格式）
- **replies_llm_cfg.json**: 网友回复生成配置（2-40楼，跳过11楼、21楼、31楼、41楼，结局方向开放多元）
- **update_llm_cfg.json**: 楼主回复生成配置（11楼、21楼、31楼、41楼，结局内容分散到多楼，结局类型开放多元，有深度）

## 状态定义
- **GlobalState**: 包含所有节点的中间状态
- **GraphInput**: 输入参数（主题）
- **GraphOutput**: 输出参数（完整的论坛体小说）
- **OutlineInput/Output**: 故事大纲节点的出入参
- **OpeningInput/Output**: 开篇帖子节点的出入参
- **RepliesInput/Output**: 网友回复节点的出入参
- **UpdateInput/Output**: 楼主回复节点的出入参
- **MergeInput/Output**: 故事整合节点的出入参
